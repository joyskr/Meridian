## Global Rules

- Organization context is derived from authenticated session state
- No business endpoint trusts organization identifiers from the frontend as authority input
- All externally referenced business entity IDs are opaque immutable public identifiers generated by the backend
- Internal database identifiers must never be exposed through API or public tracking surfaces
- Tracking tokens are assignment-owned routing identities and are not interchangeable with public record identifiers
- Public endpoints are limited to authentication challenges, click ingest, and conversion ingest
- All write endpoints require validation, authorization, and tenant ownership checks
- Error shape, pagination, sorting, and filtering must stay consistent across modules
- Public tracking redirects use assignment-owned trusted redirect configuration only

## Authentication Classes

- `Public`

- `Platform Admin`

## RBAC Baseline

- Fixed MVP roles:
  - `owner`
  - `admin`
  - `manager`
  - `analyst`
  - `viewer`
- Membership statuses:
  - `active`
  - `deactivated`
- Tenant access is granted only through:
  - authenticated session
  - active membership
- `owner` and `admin` are the organization-administration roles for MVP
- `owner`-only actions:
  - assigning `owner`
  - assigning `admin`
  - demoting or deactivating `owner` or `admin`
  - future billing-related organization settings
- Billing visibility baseline:
  - `owner`: allowed
  - `admin`: allowed
  - all other org roles: denied
- Billing mutation baseline:
  - `Platform Admin` only until customer-facing billing workflow exists
- Audit log visibility baseline:
  - `owner`: allowed
  - `admin`: allowed
  - all other org roles: denied
- Custom roles are intentionally excluded from MVP
- Runtime-configurable role inheritance is intentionally excluded from MVP

### `POST /memberships/provision-user`

- Authentication: `Organization Admin`
- Behavior:
  - requires an active organization selection on the session
  - creates a work-account user directly in the active organization
  - creates a membership immediately
  - marks the work-account email as verified at creation time
  - issues a one-time password-setup challenge for first login
- Request contract:
  - `email`: string, normalized email
  - `role`: one of `owner`, `admin`, `manager`, `analyst`, `viewer`
  - `manager_membership_id`: string or `null`
- Validation rules:
  - `owner` may provision any role
  - `admin` may provision only `manager`, `analyst`, or `viewer`
  - no provisioning flow may assign a role higher than the acting membership may assign
  - email must not already exist on any Meridian user
  - `manager_membership_id` may be set only when provisioning `analyst` or `viewer`
  - `manager_membership_id` must resolve to an active `manager` membership in the same organization
- Response contract:
  - `user.id`
  - `user.email`
  - `user.email_verified`
  - `user.created_at`
  - `membership.id`
  - `membership.role`
  - `membership.status`
  - `membership.joined_at`
  - `membership.manager`
  - `password_setup.required`
  - `password_setup.token`
  - `password_setup.expires_at`
- Status: implemented

### `PATCH /memberships/{membership_id}/role`

- Authentication: `Organization Admin`
- Request contract:
  - `role`: one of `owner`, `admin`, `manager`, `analyst`, `viewer`
- Validation rules:
  - active organization selection is required
  - target membership must belong to the active organization
  - self-role change is forbidden
  - only `owner` may assign `owner` or `admin`
  - `admin` may manage only `manager`, `analyst`, and `viewer`
  - the last active owner cannot be demoted
- Response contract:
  - `membership.id`
  - `membership.role`
  - `membership.status`
  - `membership.joined_at`
  - `membership.manager`
  - `membership.manager.membership_id`
  - `membership.manager.user_id`
  - `membership.manager.email`
  - `user.id`
  - `user.email`
  - `user.email_verified`
  - `current`
- Status: implemented

### `PATCH /memberships/{membership_id}/manager`

- Authentication: `Organization Admin`
- Request contract:
  - `manager_membership_id`: string
- Validation rules:
  - active organization selection is required
  - target membership must belong to the active organization
  - target membership must not be the current actor
  - only `analyst` or `viewer` memberships may be assigned to a manager
  - target membership must be active
  - `manager_membership_id` must resolve to an active `manager` membership in the same organization

- Response contract:
  - `membership.id`
  - `membership.role`
  - `membership.status`

  - `current`
- Status: implemented

## Conversion Architecture Freeze

- Conversion public IDs use the opaque immutable `cnv_*` strategy
- Internal database identifiers must never be exposed through conversion APIs
- Attribution ownership chain is frozen as:
  - tracking token
  - assignment
  - click
  - conversion
- `click_id` may be used as a direct lookup key where applicable, but only as a Meridian-issued tracking artifact inside the same attribution chain
- Caller-supplied organization, publisher, advertiser, offer, or assignment identifiers must never be trusted as attribution authority
- One click may legitimately produce many conversions
- Conversion dedupe must therefore be based on advertiser-scoped external event identity or equivalent idempotency key, not on a one-click-to-one-conversion assumption
- Minimum MVP conversion lifecycle states:
  - `received`
  - `finalized`
  - `rejected`
- Minimum MVP conversion timestamps:
  - `occurred_at`
  - `received_at`
  - `finalized_at`
- `received_at` is mandatory
- `occurred_at` should be captured when supplied by the advertiser integration
- `finalized_at` is set only when attribution and financial outcomes are locked
- Approved MVP event-family review:
  - `lead`
  - `registration`
  - `install`
  - `deposit`
  - `sale`
  - `custom`
- The same click may legitimately produce:
  - multiple different event types
  - repeated occurrences of the same event type when advertiser event identity distinguishes them
- Supported public conversion sources in MVP:
  - advertiser-side postbacks
  - legacy public `/gpixel`
  - legacy public `/goal`
- `/gpixel` and `/goal` remain separate public endpoint surfaces in MVP
- `/gpixel` and `/goal` must be thin transport aliases over one shared conversion-ingestion service
- Conversion source is not attribution authority and does not override the tracking-derived attribution chain
- Minimum click lookup order:
  1. exact `click_id`
  2. exact `sub1`
  3. exact `sub2`
  4. exact `sub3`
  5. exact `sub4`
  6. exact `sub5`
- If `click_id` is supplied and does not resolve, lower-priority `sub1`-`sub5` lookup is not attempted
- In that case the conversion is rejected as `click_not_found`
- Conversion dedupe uniqueness scope:
  - primary: `advertiser_id + external_event_id`
  - fallback when `external_event_id` is absent: `advertiser_id + idempotency_key`
- `source_surface` is not part of the dedupe uniqueness scope
- All public conversion-entry surfaces require `advertiser_id` as the minimum source-scoping input in MVP
- `utm_*` values are not attribution lookup keys in MVP
- If multiple lookup inputs are supplied and they contradict one another, the conversion is rejected as `attribution_conflict`
- If one `sub1`-`sub5` lookup input resolves to multiple clicks within the scoped advertiser, the conversion is rejected as `attribution_conflict`
- `event_type` maps to the attributed offer's `event_code`
- Unknown `event_type` values are rejected and are not silently ignored
- Minimum rejection or failure code inventory:

## Payout Architecture Freeze

- `PayoutBatch` owns many `Payout` rows
- Payout rows are conversion-granular, not aggregated
- One finalized conversion may appear in at most one persisted payout row
- One payout row represents exactly one finalized conversion
- Rejected conversions are ineligible for payout rows
- Non-finalized conversions are ineligible for payout rows
- Once persisted, payout rows are immutable financial history

### Payout batch lifecycle

- Minimum MVP states:
  - `draft`
  - `approved`
  - `exported`
  - `reconciled`
- Allowed forward transitions:
  - `draft -> approved`
  - `approved -> exported`
  - `exported -> reconciled`
- Forbidden transitions:
  - no backward transitions
  - no direct `draft -> exported`
  - no direct `draft -> reconciled`
  - no direct `approved -> reconciled`
- Draft batches may be deleted before approval
- Approved batches are locked
- Exported batches are locked
- Reconciled batches are permanently locked

### Batch creation workflow

- MVP batch creation is manual only
- Automatic scheduled batches are excluded from MVP
- `POST /payout-batches/preview`
  - preview only
  - does not persist data
  - does not reserve conversions
- `POST /payout-batches`
  - persists the batch and payout rows atomically
  - snapshots payout-row data permanently

### Payout-row snapshot boundary

- Mandatory identity snapshot:
  - `payout_id`
  - `batch_id`
  - `conversion_id`
  - `click_id`
  - `offer_id`
  - `offer_assignment_id`
  - `publisher_id`
  - `advertiser_id`
- Mandatory financial snapshot:
  - `advertiser_payout`
  - `publisher_payout`
  - `publisher_payout_source`
  - `publisher_tier`
  - `publisher_tier_percent`
  - `assignment_override_amount`
- Mandatory conversion-fact snapshot:
  - `event_type`
  - `source_surface`
  - `finalized_at`

### Duplicate-payment protection

- One finalized conversion may belong to zero or exactly one persisted payout row
- One finalized conversion cannot be paid twice
- One payout row cannot belong to multiple batches
- Duplicate batch creation must exclude conversions already represented by persisted payout rows
- Minimum payout-safety uniqueness boundary:
  - one persisted payout row per `conversion_id`
  - enforced by a database-level uniqueness constraint on `payouts.conversion_id`
- Exported and reconciled batches preserve payout history unchanged

### Interaction with conversion reprocessing

- If a conversion is not yet in any payout batch:
  - reprocessing is allowed
  - future payout values follow the latest finalized conversion snapshot
- If a conversion belongs to a draft batch:
  - conversion reprocessing is blocked
  - operator must delete the draft batch first if they need to rerun payout eligibility
- If a conversion belongs to an approved batch:
  - conversion reprocessing is blocked
  - later manual payout adjustment flow must handle corrections
- If a conversion belongs to an exported batch:
  - conversion reprocessing is blocked
  - later manual payout adjustment flow must handle corrections
- If a conversion belongs to a reconciled batch:
  - conversion reprocessing is blocked
  - later manual payout adjustment flow must handle corrections

### Export model

- Export does not change financial truth
- Export is an operational batch status only
- Repeated export attempts may be idempotent and regenerate the same output
- Export files themselves are not persisted in MVP

### Reconciliation model

- Full-batch only in MVP
- No partial reconciliation
- No payout reversals
- No negative payouts
- No accounting ledger or balance engine

### RBAC

- `owner`
  - payout reads: allowed
  - batch creation: allowed
  - approval: allowed
  - export: allowed
  - reconciliation: allowed
- `admin`
  - payout reads: allowed
  - batch creation: allowed
  - approval: allowed
  - export: allowed
  - reconciliation: allowed
- `manager`
  - payout reads: allowed
  - batch creation: denied
  - approval: denied
  - export: denied
  - reconciliation: denied
- `analyst`
  - payout reads: allowed
  - batch creation: denied
  - approval: denied
  - export: denied
  - reconciliation: denied
- `viewer`
  - payout reads: denied
  - batch creation: denied
  - approval: denied
  - export: denied
  - reconciliation: denied

### Historical immutability

- Meridian MVP preserves payout rows permanently once a batch is approved
- Meridian MVP does not overwrite payout rows
- Future correction paths must use deletion of unapproved draft batches or later adjustment mechanisms rather than row mutation

### Audit requirements

- Audit logging is required for:
  - batch creation
  - draft batch deletion
  - batch approval
  - batch export
  - batch reconciliation
  - manual payout adjustments when implemented
  - any operation affecting persisted payout history
